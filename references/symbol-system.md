# PrismGrid Symbol System

PrismGrid is the original non-character visual identity for Article Visual Grammar. It creates recognition through reusable semantic marks, not through a mascot, figure, face, or fixed story world.

Chinese name: `棱格文图语法`.

Internal mechanism: `Anchor Glyph System`. PrismGrid is the brandable system; Anchor Glyphs are the semantic marks inside it.

Use it when the user wants consistent article illustrations that remain clear across many topics.

## North Star

The system should feel like:

- an editorial diagram came alive just enough to be memorable
- an instruction manual with better taste
- a field atlas for ideas
- a reasoning path made visible

It should not feel like:

- a mascot illustration
- a whiteboard doodle
- a PPT icon set
- a generic AI infographic
- a copy of Xiaohei or any character-based skill

## Core Visual DNA

- Off-white or pale paper background.
- Clean black or dark ink line, not sketchy.
- One visible carrier path, spine, ribbon, shelf, or frame.
- Concrete object symbols built from simple geometry.
- Structured label containers, not floating text.
- 1-3 semantic colors per image.
- Calm grid discipline with one small visual twist.
- No faces, eyes, limbs, mascot bodies, or recurring character.
- No head-and-shoulders user pictograms. Represent humans through role cards, empty seats, decision desks, audience slices, or target fields.

## Primitive Geometry

Build all PrismGrid marks from five primitive forms:

- rounded rectangle
- circle / dot
- arc
- wedge
- straight line / polyline

Allowed modifiers:

- notch
- tab
- cut corner
- bracket
- split
- stack
- seal

Rules:

- Do not add a new primitive unless the article literally requires a domain-specific object.
- Use 8px or 16px corner radius feeling; do not mix many radius styles.
- Prefer flat 2D construction. No glossy 3D, glass, neon, bokeh, or cinematic lighting.
- Keep object silhouettes readable at thumbnail size.

## Symbol Classes

Use 3-7 symbol classes per image. Do not use all at once.

### 1. Anchor Node

Meaning: a main idea, state, stage, or decision point.

Form:

- rounded card
- clipped-corner tile
- circular seal
- tabbed folder
- heavy block
- framed specimen card

Rules:

- Use 3-5 anchor nodes for multi-part images.
- Make the focal anchor 1.2-1.6x larger than supporting anchors.
- Put L1 labels on or inside anchor nodes.

### 2. Carrier

Meaning: the reading path and hierarchy spine.

Form:

- ribbon
- ruled line
- rail
- conveyor strip
- shelf
- river band
- route line
- central spine

Rules:

- Carrier must be visible within one second.
- Carrier enters from left/top and exits right/bottom unless the concept requires a loop.
- Anchor nodes attach to the carrier; they do not float.

### 3. Boundary Mark

Meaning: permission, filter, trust, safety, threshold, acceptance, rejection.

Form:

- gate frame
- membrane line
- slot
- checkpoint bracket
- stamped seal
- cut line
- lock tab

Rules:

- Show one thing crossing, blocked, or transformed.
- Do not use a generic lock icon unless the article is literally about locks.

### 4. Container Mark

Meaning: memory, grouping, ownership, archive, context, category.

Form:

- tray
- drawer
- shelf
- jar
- folder
- capsule
- labeled box
- specimen case

Rules:

- Containers must have labels.
- Use containment to show structure, not decoration.

### 5. Evidence Mark

Meaning: proof, repeated observation, source, receipt, data point.

Form:

- stacked slips
- ticked card
- pinned sample
- small repeated dots
- ledger row
- numbered tag
- micro chart

Rules:

- Evidence should be countable or inspectable.
- Repeat same-scale marks when quantity matters.

### 6. Tension Mark

Meaning: tradeoff, conflict, pressure, mismatch, unstable balance.

Form:

- split tile
- scale tray
- opposing brackets
- compressed spring
- cracked seam
- tilted platform
- pressure band

Rules:

- Show which force acts on which object.
- Avoid dramatic explosions; use local pressure.

### 7. Transformation Mark

Meaning: process, change, synthesis, distillation, repair.

Form:

- filter
- press
- lens
- hinge
- patch
- oven slot
- funnel only when physically specific
- before/after joined tile

Rules:

- Show input, action, and output.
- One transformation per image unless the image is a pipeline.

### 8. State Mark

Meaning: pass, fail, risk, done, wait, draft, verified.

Form:

- small stamp
- corner tab
- badge
- check notch
- warning triangle
- status bead
- colored pin

Rules:

- Use L3 text here.
- State marks must attach to an anchor or object.

### 9. Human-Stake Mark

Meaning: human consequence without drawing a person.

Form:

- empty chair
- hand-sized note
- pulse line
- coffee ring
- held space outline
- decision desk
- calendar pressure mark
- audience slice card
- role tab
- target field

Rules:

- Use sparingly.
- Do not turn it into a character scene.
- Do not use head-and-shoulders icons, silhouettes, user avatars, stick figures, or profile heads.
- If the concept is "user", prefer an audience slice card, target field, role tab, or empty seat.

## Semantic Shape Rules

Within one image:

- Cards = claims or states.
- Slips = evidence or input.
- Tabs = status or ownership.
- Lines = relation or path.
- Frames = boundaries or context.
- Containers = memory or grouping.
- Dots = quantity or repeated cases.
- Cracks = failure or mismatch.
- Ribbons = sequence or argument flow.
- Seals = decision, verification, rejection, or risk.

Do not let the same shape mean two unrelated things in one image.

## Line System

Use three line weights:

- Hairline: grid, chart axis, secondary callout. Approx. 1 unit.
- Relation line: object-to-object relation. Approx. 1.5 units.
- Carrier line: main reader path. Approx. 2.5 units.

Rules:

- Carrier line is darkest or thickest.
- Relation lines are shorter than carrier lines.
- Callout lines touch the object they label.
- Do not use arrow soup. Use arrowheads only when direction cannot be read from layout.
- Rounded line endings feel editorial; square line endings feel technical. Pick one per image.
- Use exactly one dominant carrier line per image.

## Color System

Default palette:

- Ink: near-black for outlines and key text, `#17202A` feeling.
- Paper: off-white or pale warm background, `#F7F4EE` feeling.
- Quiet: light gray for grid, inactive objects, or context, `#D7D9D6` feeling.
- Passage: muted amber for movement, sequence, or active path, `#D9A441` feeling.
- Evidence: muted blue / cyan for sources, feedback, memory, or data, `#3D7D90` feeling.
- Risk: muted red for failure, rejection, pressure, or warning, `#C84B44` feeling.
- Growth: muted green for learning, recovery, or accumulation, `#668A5A` feeling.

Rules:

- Use Ink, Paper, and Quiet as the neutral base.
- Pick either Passage or Evidence as the main accent for most images.
- Add Risk only when risk is part of the article idea.
- Use Growth only when growth, repair, or accumulation is actually the point.
- Use 1 dominant semantic color, 1 supporting color, and 1 alert color max.
- Color carries meaning; it is not decoration.
- Keep the same color meaning across an article package.
- Avoid the fixed red/orange/blue handwriting look associated with Xiaohei-like work.

## Label Containers

Text must live in a container.

Allowed:

- anchor label strip
- specimen caption
- tray label
- route tag
- state stamp
- side note box
- corner tab
- callout flag
- panel caption

Rules:

- L1 label goes on the focal anchor.
- L2 labels go inside or beside anchor nodes.
- L3 marks go on stamps, tabs, or badges.
- Chinese: 8 characters max per container by default.
- English: 3 words max per container by default.
- Container padding should feel consistent across the image.
- Do not use free-floating paragraphs.
- Do not use a top-left title unless requested.
- Do not add a visible legend panel by default. The package symbol legend is an internal consistency contract, not something the viewer must read.
- If a visible legend is necessary, keep it to 2-3 marks and place it as a small side note, not a large key box.

## Layout Skeletons

Choose one skeleton before prompting.

### Spine Map

Best for: argument structure, multi-step reasoning, analysis.

Visual: one central spine with 3-5 anchor nodes and side notes.

### Shelf Sequence

Best for: timelines, phases, process, before/after.

Visual: one shelf or rail holds stages in order.

Risk: can become a corporate process slide.

Use only when sequence is the core idea. Add one editorial asymmetry: one enlarged end state, one folded artifact, one local failure mark, or one embedded micro chart. Do not put checkmarks under every node.

### Gate Transfer

Best for: permission, trust, security, review, validation.

Visual: outside object -> boundary mark -> changed inside object.

### Atlas Spread

Best for: categories, comparisons, observations.

Visual: specimen cards on a controlled grid with notes and legend.

### Cutaway Block

Best for: hidden layers, mechanisms, system internals.

Visual: one object opened into labeled compartments.

### Pressure Scale

Best for: tradeoffs, incentives, risk, priorities.

Visual: two trays, opposing brackets, or compressed platform.

### Loop With Residue

Best for: iteration, feedback, compounding, learning.

Visual: loop carrier with visible accumulated trace.

## Micro Chart Rules

Use only four chart forms inside PrismGrid:

- bar
- dot
- step
- quadrant

Rules:

- Charts must embed inside an anchor node, side note, carrier stop, or cutaway.
- No standalone chart pasted beside the illustration.
- No 3D chart, pie chart, gauge dashboard, or decorative sparkline.
- The chart must answer one concrete question: more/less, before/after, threshold, distribution, or quadrant choice.
- Chart labels obey the same text-container rules.
- Keep micro charts under 20% of the image area.
- Do not give micro charts full chart titles unless the chart is the focal object.
- Use chart marks as evidence embedded in the system, not as a separate dashboard.

## Package Consistency

For a multi-image article package:

- Keep background and ink color constant.
- Keep line endings consistent.
- Keep label containers from the same family.
- Reuse 2-3 symbol classes across all images.
- Vary layout skeletons so the article does not feel repetitive.
- Keep color semantics stable.
- Keep primitive geometry stable: five primitives, one radius family, one line ending style.
- Keep at least one recurring PrismGrid mark across the package, such as state stamp, evidence slip, or anchor label strip.

## Package Symbol Legend

Before planning multiple images, define one shared legend:

```text
package symbol legend:
shared primitive geometry:
shared shape meanings:
shared color meanings:
shared label containers:
recurring marks:
inheritance rule:
```

Example:

```text
shared primitive geometry: rounded rectangle, dot, line, wedge
shared shape meanings: rounded card = claim/state; slip = evidence; seal = validation/risk
shared color meanings: amber = active path; blue = evidence/memory; red = failure boundary
shared label containers: anchor strip, state stamp, side note
recurring marks: evidence slip + route tag appear in every image
inheritance rule: every image reuses the same label containers and color meanings; layout skeleton may vary
```

The package legend is the non-character equivalent of an IP bible. It keeps recognition stable without a mascot.

Do not render the full package legend as a visible legend box by default. Viewers should understand the image through structure, not by reading a map key. Render only the marks that the current idea needs.

## Anti-Courseware Rules

PrismGrid should not look like a training slide.

Avoid:

- large visible legend box
- numbered badge on every node unless numbering is the idea
- checkmark under every step
- bottom "principle bar" summarizing the whole image
- long explanatory footer
- equal-sized cards in a perfectly sterile row
- icon + label repeated as a UI menu

Prefer:

- one focal node 1.2-1.6x larger than others
- one local failure mark or transformation moment
- one embedded artifact slip or micro chart
- quiet margins and uneven-but-aligned composition
- a carrier that bends, loops, gates, or shelves the idea instead of merely connecting boxes

## Forbidden

- faces, eyes, limbs, mascot bodies
- head-and-shoulders user pictograms, profile heads, silhouettes, stick figures
- black blob characters
- cute workers or tiny helpers
- generic robot, brain, rocket, lightbulb
- whiteboard doodle identity
- casual handwritten labels everywhere
- decorative arrows
- unlabelled floating symbols
- random icon packs
- too many primitive shapes
- multiple carrier lines competing for attention
- large visible legend boxes by default
- checkmarks under every workflow step
- bottom principle bars or long footer summaries
- chart junk
- dense symbolic collage with no carrier

## QA

Accept the symbol system if:

- the image is recognizable without any character
- each repeated shape has a stable meaning
- the carrier creates a clear reading path
- labels live in containers
- color has semantic purpose
- the system can be reused for a different article without becoming a mascot
- the graphic still works as a thumbnail

Reject or regenerate if:

- the strongest identity is a person-like figure
- a "user" becomes a head-and-shoulders avatar instead of a role/target/seat mark
- the marks feel like unrelated icons
- the image needs a paragraph to explain the symbols
- the palette is decorative rather than semantic
- shapes change meaning inside one image
- the result looks like a PPT icon diagram
- micro charts float outside the structure
- the image reads as a generic process slide or training handout
